Quickstart Guide for New Contributors

Overall Goal

The NFT Monitoring project is a Vite/React dashboard that consumes live hydroponic sensor data through STOMP/WebSocket and REST APIs. It renders the data across multiple pages (Overview, Live, Reports, Cameras, and Sensor Configuration).

Structure and Main Flow

  • Entry point: src/index.jsx boots the app with ReactDOM and wraps everything in SensorConfigProvider so sensor values and ranges remain editable.
  • Routes: src/App.jsx defines routing; all pages live under MainLayout, which provides the sidebar and ReportsFiltersProvider before rendering content via <Outlet />.
  • Pages: The src/pages folder contains each page (e.g., Overview, Live, Reports). Pages reuse shared components in src/components and utilities in src/utils.
  • Live data: The src/hooks/useStomp.js hook builds a shared STOMP client, auto-subscribes to topics, and manages connection/reconnection so each page only defines its topics and message handlers.
  • Configuration and API: src/context/SensorConfigContext.jsx loads/updates sensor configuration through REST methods in src/api/sensorConfig.js. Values for VITE_API_BASE_URL and WebSocket addresses are set in .env.
  • Testing: Vitest specs live in tests and use helpers like tests/utils/renderWithProviders to render components with required contexts.

Key Notes Before Development

  • Most pages assume they are rendered inside MainLayout so the sidebar and report filters are available.
  • Do not bypass useStomp; control subscriptions through its topics array.
  • Set environment variables for WebSocket and REST in .env to target the correct server.
  • When creating new pages, leverage components in src/components and src/pages/common to keep style and UX consistent.

Suggested Paths for Learning More

  1. Data normalization: The helpers in src/pages/Overview/utils show how raw sensor data becomes cards and charts.
  2. Report filtering: Explore src/pages/Reports/context/ReportsFiltersContext.jsx and related components to see how ranges and composite IDs are selected.
  3. Testing patterns: The tests/mocks and tests/utils folders offer examples of API mocking and rendering with providers.
